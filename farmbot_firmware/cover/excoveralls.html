    18 The firmware starts in a `:transport_boot` state, moving to `:boot`. It then

    22 State machine flows go as follows:
    23 ## Boot
    24 :transport_boot
    25 |> :boot
    26 |> :no_config
    27 |> :configuration
    28 |> :idle
    29
    30 ## Idle
    31 :idle
    32 |> :begin
    33 |> :busy
    34 |> :error | :invalid | :success
    35
    36 # Constraints and Exceptions
    37 Commands will be queued as they received with some exceptions:
    38 * if a command is currently executing (state is not `:idle`),
    39 proceding commands will be queued in the order they are received.
    40 * the `:emergency_lock` and `:emergency_unlock` commands go to the front
    41 of the command queue and are started immediately.
    42 * if a `report_emergency_lock` message is received at any point during a
    43 commands execution, that command is considered an error.
    44 (this does not apply to `:boot` state, since `:parameter_write`
    45 is accepted while the firmware is locked.)
    46 * all reports outside of control flow reports (:begin, :error, :invalid,
    47 :success) will be discarded while in `:boot` state. This means while
    48 boot, position updates, end stop updates etc are ignored.
    49

    51 GCODES should be exchanged in the following format:

    54 * `command` - either a `RXX`, `FXX`, or `GXX` code.
    55 * `args` - a list of arguments to be processed.

    58 {"123", {:report_some_information, [h: 10.00, u: 90.10]}}

    60 {"555", {:fire_laser, [w: 100.00]}}
